## PROJECT OVERVIEW

Project Name: EventHub
Purpose: A comprehensive platform for hosting and managing college events. Provides centralized solution for students and administrators to create, discover, register, and manage various college activities.

## ALL API ROUTES

METHOD /full/path → Controller Function → Middleware Used → Description

─ AUTHENTICATION ROUTES (/api/v1/auth):
POST   /api/v1/auth/signup        → registerUser()          → (none)              → User registration - creates new account with firstname, lastname, email, collegename, password. Auto-login after registration.
POST   /api/v1/auth/login         → loginUser()             → (none)              → User login - verifies email and password, returns JWT tokens in cookies
POST   /api/v1/auth/logout        → logoutUser()            → authMiddleware      → User logout - clears refresh token and cookies
GET    /api/v1/auth/me            → getMe()                 → authMiddleware      → Fetch current authenticated user profile

─ EVENT ROUTES (/api/v1/events):
POST   /api/v1/events/            → Createevent()           → authMiddleware, authorized('Organiser', 'Faculty'), validate(createeventschema) → Create new event (Organizer/Faculty only). Requires name, type, date. Optional: time, venue, maxTeams, prize, description
GET    /api/v1/events/            → getevent()              → authMiddleware      → Get all events for user's college (filtered by collegename, sorted by newest first)
GET    /api/v1/events/:id         → getsingleevent()        → authMiddleware      → Get single event details by ID
PUT    /api/v1/events/:id         → updateevent()           → authMiddleware, authorized('Organiser', 'Faculty') → Update event details by ID (title, date, time, status, prize, etc.)
DELETE /api/v1/events/:id         → deleteevent()           → authMiddleware, authorized('Organiser', 'Faculty') → Delete event by ID (removes all associated data)

─ TEAM/REGISTRATION ROUTES (/api/v1/teams):
POST   /api/v1/teams/register     → registerteam()          → authMiddleware, validate(registerevent) → Register team for an event. Input: teamname, leadname, members (array), collegeid, email, eventId. Increments event.teams count
GET    /api/v1/teams/:id/teams    → getTeams()              → authMiddleware, authorized('Organiser', 'Faculty') → Get all teams registered for specific event
GET    /api/v1/teams/registrations/mine → studentownregistration() → authMiddleware → Get student's own registrations with populated event details (name, type, status, date, venue, prize)
DELETE /api/v1/teams/:tid         → cancleregistration()    → authMiddleware, authorized('Organiser', 'Faculty') → Cancel team registration. Updates event.teams count. Owner or Organizer/Faculty can cancel

─ SCHEDULE ROUTES (/api/v1/schedule):
POST   /api/v1/schedule/:eventId  → addslot()               → authMiddleware, authorized('Organiser', 'Faculty'), validate(Schedulevalidator) → Add time slot for event. Input: title, starttime, endtime (optional), description (optional), order (optional)
GET    /api/v1/schedule/:eventId  → getschedule()           → authMiddleware      → Get all schedule slots for event (sorted by order, then starttime)
PUT    /api/v1/schedule/:eventId/:slotId → updateslot()     → authMiddleware, authorized('Organiser', 'Faculty'), validate(Schedulevalidator) → Update schedule slot details
DELETE /api/v1/schedule/:eventId/:slotId → deleteslot()     → authMiddleware, authorized('Organiser', 'Faculty') → Delete specific schedule slot

─ USER ROUTES (/api/v1/user):
GET    /api/v1/user/me            → getuser()               → authMiddleware      → Get current user's full profile
PUT    /api/v1/user/updateme      → updateme()              → authMiddleware      → Update user profile (firstname, lastname, collegename)
PUT    /api/v1/user/updatepassword → updatepassword()       → authMiddleware      → Change password. Requires oldpassword and newpassword. Verifies old password first

─ TEST ROUTE:
GET    /test                       → (inline)               → (none)              → Server health check - returns welcome message

## DATABASE MODELS

─ USER MODEL (Collection: "Users")
Fields with types and constraints:
  • firstname: String, required, trimmed
  • lastname: String, optional, trimmed
  • email: String, required, unique, lowercase, trimmed, indexed
  • collegename: String, required, trimmed
  • role: String, enum ["Student", "Organiser", "Faculty"], default: "Student"
  • password: String, required, not selected by default, hashed with bcrypt before save
  • refreshToken: String, optional, not selected by default (hidden field)
  • resetPasswordToken: String, optional (for password reset feature - unused)
  • resetPasswordExpiry: Date, optional (for password reset feature - unused)
  • createdAt: Date (Mongoose timestamp)
  • updatedAt: Date (Mongoose timestamp)

Methods:
  • isPasswordCorrect(password): Compares provided password with hashed password using bcrypt
  • generateAccessToken(): Creates JWT with _id, email, role. Signs with ACCESS_TOKEN_SECRET
  • generateRefreshToken(): Creates JWT with _id, email, role. Signs with REFRESH_TOKEN_SECRET
  • generateTemporaryToken(): Generates crypto token for password reset (unused endpoint)

Relationships: User created many Events, Teams, and Schedules

─ EVENT MODEL (Collection: "Event")
Fields with types and constraints:
  • name: String, required, trimmed
  • type: String, enum ["Hackathon", "Quiz", "Cultural", "Techtalk", "Sports"], required
  • status: String, enum ["Upcoming", "Live", "Completed"], required
  • date: Date, required
  • time: String, optional
  • venue: String, optional, trimmed
  • maxTeams: Number, optional (capacity limit for registrations)
  • prize: String, optional (prize amount/description)
  • description: String, optional
  • teams: Number, default: 0 (tracks number of registered teams)
  • collegename: String, required, trimmed, lowercase (for college filtering)
  • createdBy: ObjectId, ref: "User" (who created the event)
  • createdAt: Date (Mongoose timestamp)
  • updatedAt: Date (Mongoose timestamp)

Relationships: One Event has many Teams registrations, many Schedule slots

─ TEAM REGISTRATION MODEL (Collection: "Teams")
Fields with types and constraints:
  • teamname: String, required
  • leadname: String, required (team leader name)
  • members: Array of Strings, required, validated length 1-5
  • collegeid: String, required
  • email: String, required
  • event: ObjectId, ref: "Event" (which event is registered for)
  • registeredBy: ObjectId, ref: "user" (NOTE: Has model name mismatch - refers to "user" but Users collection is "Users")
  • status: String, enum ["Confirmed", "Cancelled"], default: "Confirmed"
  • createdBy: ObjectId, ref: "User" (deprecated, not used)
  • createdAt: Date (Mongoose timestamp)
  • updatedAt: Date (Mongoose timestamp)

Relationships: Many Teams belong to one Event, tracked by registeredBy user

─ SCHEDULE MODEL (Collection: "Schedule")
Fields with types and constraints:
  • title: String, required, trimmed
  • starttime: String, required (time format - no specific format enforcement)
  • endtime: String, optional
  • description: String, optional, trimmed
  • event: ObjectId, ref: "Event", required (which event this schedule belongs to)
  • createdBy: ObjectId, ref: "User", required (who created this schedule)
  • order: Number, optional (sequence order for schedule items)
  • createdAt: Date (Mongoose timestamp)
  • updatedAt: Date (Mongoose timestamp)

Relationships: Many Schedule slots belong to one Event

## MIDDLEWARE

1. authMiddleware (/src/middleware/auth.middleware.js)
   - Verifies JWT token from cookies.accessToken OR Authorization header (Bearer token)
   - Returns 401 if no token provided
   - Checks if token is valid using ACCESS_TOKEN_SECRET
   - Fetches user from database and makes available as req.user
   - Handles TokenExpiredError, JsonWebTokenError, and generic errors
   - Attached to all protected routes requiring authentication
   - Does NOT validate token expiry with refresh token (missing token rotation)

2. authorized(...role) (/src/middleware/role.middleware.js)
   - Role-based access control middleware (variadic function)
   - Checks if req.user.role is in allowed roles array
   - Returns 403 Forbidden if role not authorized
   - Accepted roles: "Student", "Organiser", "Faculty"
   - Used for route protection: endpoints restricted to Organiser/Faculty
   - Can be chained with authMiddleware

3. validate(schema) (/src/middleware/validate.js)
   - Joi schema validation middleware
   - Validates req.body against provided Joi schema
   - Returns 400 with first validation error if validation fails
   - Calls next() if validation passes
   - Applied to: create event, team registration, schedule creation/update

## VALIDATION (Joi Schemas)

1. createeventschema - Validates event creation request body:
   • name: string, min 3 characters, required
   • type: string, must be one of: "Hackathon", "Quiz", "Cultural", "Techtalk", "Sports", required
   • status: string, must be one of: "Upcoming", "Live", "Completed", optional
   • date: date object, required
   • time: string, optional
   • venue: string, optional
   • maxTeams: number, minimum 1, optional
   • prize: string, optional
   • description: string, optional

2. registerevent - Validates team registration request body:
   • teamname: string, min 3 characters, required
   • leadname: string, min 2 characters, required
   • members: array of strings (each min 2 chars), min 1 item, max 5 items, required
   • collegeid: string, required
   • email: string, valid email format, required
   • eventId: string, UUID format, required

3. Schedulevalidator - Validates schedule slot creation/update:
   • title: string, min 4 characters, required
   • starttime: string, required (no format validation - accepts any string)
   • endtime: string, optional
   • description: string, optional
   • order: number, optional

Note: POST /api/v1/auth/signup and /api/v1/auth/login are NOT validated in middleware. Validation occurs in controller logic only.

## UTILITY FUNCTIONS

1. generateTokens(userId) (/src/utils/generateTokens.js)
   Purpose: Generate both access and refresh JWT tokens for authenticated user
   Input: userId (MongoDB ObjectId)
   Output: { accessToken, refreshToken }
   Process:
   - Fetches user from database
   - Calls user.generateAccessToken() method
   - Calls user.generateRefreshToken() method
   - Saves refreshToken to user document in database
   - Returns both tokens
   Error handling: Throws error if user not found or token generation fails

## ENVIRONMENT VARIABLES

Required environment variables (in .env file - must be created):

1. MONGO_URL
   - MongoDB connection string
   - Example: mongodb://localhost:27017/eventhub
   - Used in: /src/model/db.js
   - Type: String

2. ACCESS_TOKEN_SECRET
   - Secret key for signing access tokens (JWT)
   - Should be long random string (min 32 characters)
   - Used in: User model generateAccessToken(), auth middleware verification
   - Expiry time: ACCESS_TOKEN_EXPIRY (default ~15min)
   - Type: String

3. ACCESS_TOKEN_EXPIRY
   - Expiration time for access tokens
   - Example: "15m", "1h", "7d"
   - Used in: User model generateAccessToken()
   - Type: String (JWT format)

4. REFRESH_TOKEN_SECRET
   - Secret key for signing refresh tokens (JWT)
   - Should be different from ACCESS_TOKEN_SECRET
   - Used in: User model generateRefreshToken()
   - Type: String

5. REFRESH_TOKEN_EXPIRY
   - Expiration time for refresh tokens
   - Example: "7d", "30d"
   - Used in: User model generateRefreshToken()
   - Type: String (JWT format)

6. NODE_ENV
   - Environment mode: "development" or "production"
   - Used in: Cookie security settings (secure flag depends on this)
   - Type: String
   - Default: Value not enforced (should be set)

7. GEMINI_API_KEY
   - Google Gemini AI API key for Google Generative AI
   - Currently imported but NOT USED in any endpoints
   - Used in: /src/services/ai.service.js (incomplete implementation)
   - Type: String

8. PORT
   - Server port number
   - Parsed but overridden by hardcoded value 8000 in index.js
   - Type: Number
   - Current: 8000 (hardcoded - BUG)

## POTENTIAL BUGS OR ISSUES

Security Issues:

1. [CRITICAL] Port Override Issue
   Location: /index.js line 4
   Issue: const PORT = 8000 || process.env.PORT;
   Problem: PORT always 8000, even if process.env.PORT is set to different value. Should be: process.env.PORT || 8000
   Impact: Environment configuration ignored, cannot run multiple instances on different ports
   Severity: HIGH

2. [CRITICAL] Hardcoded CORS Origin
   Location: /app.js line 17-19
   Issue: CORS only allows http://localhost:5500
   Problem: Hardcoded localhost - will fail in production, breaks on different domain
   Solution: Move to environment variable: process.env.FRONTEND_URL
   Severity: CRITICAL (production blocker)

3. [HIGH] Weak Password Requirements
   Location: /src/validators/joi.validate.js (signup validation missing)
   Issue: No password validation schema for signup endpoint
   Problem: Accepts any password including weak ones
   Solution: Add Joi password validation with min length, complexity requirements
   Severity: HIGH (security risk)

4. [HIGH] No Input Sanitization
   Issue: User inputs not sanitized before database storage
   Problem: Possible XSS/injection attacks through stored data
   Solution: Use Mongoose sanitization middleware or library
   Severity: HIGH

5. [HIGH] User Role Manipulation
   Location: /src/controller/auth.controller.js registerUser()
   Issue: User can set their own role during signup (role: role || "Student")
   Problem: User could register as "Organiser" or "Faculty" if role provided in request
   Solution: Always set role: "Student" for new users, only admins can change roles
   Severity: HIGH (authorization bypass)

6. [MEDIUM] Unrestricted Profile Update
   Location: /src/controller/user.controller.js updateme()
   Issue: Only updates firstname, lastname, collegename but user could manipulate request
   Problem: No validation that user can only update own profile
   Solution: Add checks req.user._id === userId from params
   Severity: MEDIUM

7. [MEDIUM] Model Name Mismatch
   Location: /src/model/registraton.model.js line 21
   Issue: registeredBy ref: "user" but model is exported as "Users" in user.model.js
   Problem: Populate may fail with MissingSchemaError
   Solution: Change to ref: "Users" to match exports
   Severity: MEDIUM (feature broken)

8. [MEDIUM] No Rate Limiting
   Issue: No rate limiting on login, signup, or general endpoints
   Problem: Brute force attacks possible on authentication endpoints
   Solution: Implement express-rate-limit middleware
   Severity: MEDIUM

9. [MEDIUM] User Email Exposure
   Location: /src/controller/event.controller.js getevent()
   Issue: Events don't have explicit access control beyond college
   Problem: Any user can see all events from their college (may be intended)
   Solution: Document scope of event visibility
   Severity: LOW (if intended)

10. [LOW] Typo in Event Controller
    Location: /src/controller/event.controller.js line 81
    Issue: "Internal servel error" (should be "server")
    Problem: Misleading error message
    Severity: LOW (cosmetic)

11. [LOW] Typo in Model
    Location: /src/model/registraton.model.js
    Issue: "registraton" instead of "registration"
    Problem: Confusing file naming convention
    Severity: LOW (cosmetic)

12. [MEDIUM] No Email Uniqueness in Teams
    Location: /src/controller/teams.controller.js registerteam()
    Issue: Only prevents duplicate email per event, but same email can register multiple teams
    Problem: Student could register multiple teams under same email for different events
    Solution: May be intended, but should be documented
    Severity: LOW-MEDIUM (depends on requirement)
